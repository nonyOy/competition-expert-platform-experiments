# 实验二：竞赛平台技术选型与概要设计

## 1. 技术选型
| 层级 | 技术 | 选择理由 |
| --- | --- | --- |
| 前端框架 | Vue 3 + Vite + TypeScript | 开发效率高，组件化清晰，适合课程项目快速实现 |
| UI 组件 | Element Plus | 后台系统组件完整，表格、表单、标签、弹窗可快速落地 |
| 状态管理 | Pinia | Vue 官方推荐，结构轻量，适合保存登录态和全局数据 |
| 请求库 | Axios | 接口调用成熟稳定，便于统一配置 |
| 后端框架 | Node.js + Express + TypeScript | 轻量、学习成本低，适合 REST API |
| 数据库 | MySQL 8 | 关系型数据适合赛事、专家、作品、任务和评分结构 |
| 版本控制 | Git | 支持实验过程留痕、提交记录和代码回滚 |

## 2. 系统架构
系统采用前后端分离架构。

```text
浏览器前端 Vue
    |
    | HTTP / REST API
    v
Node.js Express 后端
    |
    | SQL
    v
MySQL 数据库
```

前端负责页面展示、交互、表单校验和接口调用。后端负责登录校验、业务接口、数据查询和评分提交。数据库负责持久化保存用户、竞赛、专家、作品、任务和评分。

## 3. 模块划分
| 模块 | 说明 |
| --- | --- |
| 登录模块 | 用户登录、角色识别、前端路由跳转 |
| 赛事模块 | 赛事列表、新增赛事、状态展示 |
| 专家模块 | 专家资料维护、专业方向展示 |
| 作品模块 | 作品信息、所属赛事、评审状态 |
| 任务模块 | 评审任务分配、专家与作品关联 |
| 评分模块 | 专家评分、评语提交、任务状态更新 |
| 统计模块 | 平均分、排名、结果发布 |

## 4. 接口设计
| 接口 | 方法 | 说明 |
| --- | --- | --- |
| `/api/auth/login` | POST | 登录并返回用户角色 |
| `/api/competitions` | GET/POST | 查询和新增赛事 |
| `/api/experts` | GET | 查询专家列表 |
| `/api/works` | GET | 查询作品列表 |
| `/api/review-tasks` | GET | 查询评审任务 |
| `/api/scores` | POST | 提交评分 |
| `/api/statistics` | GET | 查询评分统计 |
| `/api/health` | GET | 服务健康检查 |

## 5. 数据流设计
1. 用户在前端登录页输入账号密码。
2. 前端调用后端登录接口，后端返回用户信息和角色。
3. 管理员端请求赛事、专家、作品、任务和统计接口，形成管理看板。
4. 专家端请求任务接口，筛选当前专家的任务。
5. 专家提交评分后，后端写入评分记录并更新任务状态。
6. 统计接口根据评分记录计算平均分、排名和完成率。

## 6. 数据库表设计
数据库核心表如下：

- `users`：保存登录账号、密码、角色和显示名。
- `competitions`：保存赛事名称、类别、状态、时间和说明。
- `experts`：保存专家姓名、职称、专业方向、联系方式和启用状态。
- `works`：保存作品标题、作者、单位、摘要和评审状态。
- `review_tasks`：保存竞赛、作品、专家、截止时间和任务状态。
- `scores`：保存评分维度、总分和评语。
- `announcements`：保存结果发布信息。

完整建表脚本见 `docs/sql/schema.sql`。

## 7. 方案评审
| 评审维度 | 说明 |
| --- | --- |
| 可实现性 | 技术栈成熟，课程实验环境容易搭建 |
