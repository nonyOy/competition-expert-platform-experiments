# 实验一：竞赛专家管理平台需求分析与 PRD

## 1. 项目背景
高校与机构在组织竞赛时，常见流程包括赛事发布、作品提交、专家邀请、评审分配、评分统计和结果发布。传统方式通常依赖表格、邮件和线下沟通，存在任务分配不清晰、评分进度难追踪、结果汇总耗时、数据留痕不足等问题。

竞赛专家管理平台面向竞赛组织方和评审专家，提供统一的赛事评审管理入口，帮助管理员完成评审过程管控，帮助专家高效查看任务、阅读作品并提交评分。

## 2. 用户角色
| 角色 | 目标 | 核心操作 |
| --- | --- | --- |
| 管理员 | 组织竞赛、维护数据、分配评审、统计结果 | 赛事管理、专家管理、作品管理、任务分配、结果发布 |
| 评审专家 | 接收任务、查看作品、提交评分意见 | 查看待评审任务、填写评分、提交评语、查看历史记录 |
| 参赛者 | 提交作品并查看结果 | 本实验聚焦管理与专家端，参赛者端作为后续扩展 |

## 3. 业务流程
1. 管理员创建竞赛，填写竞赛名称、类别、时间、说明。
2. 管理员维护专家资料，确认专家专业方向和启用状态。
3. 管理员录入参赛作品，关联对应竞赛。
4. 管理员将作品分配给专家，形成评审任务。
5. 专家登录平台，查看待评审作品。
6. 专家按创新性、实践性、展示表达三个维度评分并提交评语。
7. 管理员查看评分统计，生成排名并发布结果。

## 4. 跨端需求
- 管理端适合桌面浏览器使用，重点是数据列表、筛选、统计和任务配置。
- 专家端需适配电脑和移动端，重点是任务查看、作品阅读和评分表单。
- 页面需要响应式布局，窄屏下保持导航、表格、表单可用。

## 5. 功能需求
### 5.1 登录与角色识别
- 用户输入账号和密码登录。
- 系统根据账号角色进入管理员端或专家端。
- 登录态保存在前端本地状态中，用于页面访问控制。

### 5.2 管理员端
- 首页看板：展示赛事数量、专家数量、待评审作品、平均分、近期任务和赛事状态。
- 赛事管理：查看赛事列表，支持新增赛事、状态展示和关键词搜索。
- 专家管理：查看专家信息、专业方向、联系方式、启用状态。
- 作品管理：查看作品所属赛事、作者、单位、摘要和评审状态。
- 任务分配：查看任务、专家、作品、截止日期和处理状态。
- 统计发布：查看作品平均分、排名和结果发布信息。

### 5.3 专家端
- 评审工作台：查看待评审任务、已评审任务和截止日期。
- 作品详情：查看作品标题、作者、单位、摘要和所属竞赛。
- 评分提交：填写创新性、实践性、展示表达三项评分和综合评语。
- 评审记录：展示已提交评分与历史意见。

## 6. 非功能需求
- 易用性：操作入口清晰，列表支持搜索和状态识别。
- 可维护性：前后端分离，接口结构清晰，便于扩展。
- 可靠性：评分提交后更新任务状态，避免重复统计。
- 安全性：课程实验版本采用简单登录，生产环境需增加加密、Token、权限校验和审计日志。
- 性能：列表数据量较小时直接分页展示，后续可扩展服务端分页。

## 7. 原型结构说明
- 登录页：品牌标题、账号输入、角色说明。
- 管理端：左侧导航、顶部用户区、内容区。
- 专家端：任务列表与评分面板结合，减少跳转。
- 统计页：排名表、平均分、结果发布状态。

## 8. PRD 自查评审
| 检查项 | 结果 |
| --- | --- |
| 是否明确项目背景 | 是 |
| 是否明确用户角色 | 是 |
| 是否覆盖核心业务流程 | 是 |
| 是否包含跨端需求 | 是 |
| 是否包含功能和非功能需求 | 是 |
| 是否便于进入概要设计 | 是 |

## 9. 实验总结
通过本实验，完成了竞赛专家管理平台的需求分析和 PRD 编写。系统核心价值是将竞赛评审流程线上化、结构化和可追踪化，为后续技术选型、数据库设计和代码实现提供依据。
